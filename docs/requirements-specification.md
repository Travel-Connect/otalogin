# OTAログイン支援ツール — 要件定義書

**文書バージョン**: 1.0
**最終更新**: 2026-03-25
**対象範囲**: MVP + 拡張チャネル + 管理機能

---

## 1. プロジェクト概要

### 1.1 目的

宿泊施設が利用する複数のOTA（Online Travel Agency）管理画面へのログイン作業を効率化する。
Chrome拡張機能とWebポータルの連携により、1クリックでOTAログインページを開き、ID/PW入力を自動化する。

### 1.2 対象ユーザー

- 宿泊施設の運営スタッフ（日常的にOTA管理画面を操作する担当者）
- 施設管理者（アカウント情報の登録・更新を行う管理者）

### 1.3 システム構成

| コンポーネント | 技術 | ホスティング |
|---|---|---|
| Webポータル | Next.js 14 App Router + TypeScript | Vercel |
| DB/認証 | Supabase (PostgreSQL + Auth) | Supabase Cloud |
| Chrome拡張 | Manifest V3 | GitHub配布 |
| マスタシート | Google Spreadsheet | Google Workspace |

## 2. 機能要件

### 2.1 ログイン自動化

#### FR-001: 手動ログイン実行

- ユーザーがポータルの施設詳細画面から「ログイン実行」ボタンを押すと、対象OTAのログインページが新しいタブで開き、ID/PWが自動入力される
- 同一ウィンドウ内にタブを追加する（新ウィンドウは開かない）

#### FR-002: ログインモード

| モード | 動作 | 対象チャネル |
|---|---|---|
| 通常 (normal) | ID+PW入力→送信 | 楽天, じゃらん, ねっぱん, 一休, スカイチケット, OTS, ちゅらとく, リンカーン, るるぶ, ダイナイベ, 手間いらず, 予約プロ, tripla, CHILLNN, ミンパクイン |
| ID入力のみ (id_only) | IDのみ入力して停止 | Booking.com, Agoda, Expedia |
| ID+PW入力のみ (skip_submit) | ID+PW入力して送信しない | Trip.com |
| リンク専用 (link_only) | 公開ページURLのみ（ログイン自動化なし） | Airbnb |
| マルチステップ (multi_step) | ステップ順に入力→送信 | 楽天(SSO), るるぶ(OTP) |

#### FR-003: 安全性ルール

- ログイン実行は1ジョブにつき1回のみ（無限ループ防止）
- ジョブ実行前に `pending_job` をstorageから削除
- ASP.NET等のフルページリロードでの再実行防止

#### FR-004: ユーザー別クレデンシャル（リンカーン）

- リンカーンはユーザー別のID/PWをサポート
- ジョブ作成者のメールでアカウント検索→なければ共有クレデンシャルにフォールバック

### 2.2 施設管理

#### FR-010: ダッシュボード

- 全施設をカード形式で一覧表示
- 各カードにOTAチャネルのステータスをアイコンで表示（正常/エラー/実行中/未登録）
- 施設名での検索
- タグによるフィルタリング（URLクエリパラメータ `?tag=` で永続化＝ディープリンク対応）
- ステータスによるフィルタリング

#### FR-011: 施設詳細画面

- チャネルごとのログインID/PW表示（PWはマスク表示、目アイコンで表示切り替え）
- ログイン実行ボタン
- マスタPWと同期ボタン
- マスタに転記ボタン
- 公開ページ/管理画面のURL管理

#### FR-012: タグ管理

- 施設詳細画面でタグの追加・削除が可能
- テキスト入力 + 既存タグのサジェスト表示
- クリックでサジェスト一覧を表示
- 「タグを保存」ボタンで明示的に保存
- 全ユーザーが編集可能

#### FR-013: 施設並べ替え

- ヘッダーの「並べ替え」ボタンで並べ替えモードに入る
- ドラッグ＆ドロップで施設カードの順序を変更
- iPhoneホーム画面風の揺れアニメーション（wiggle）
- ユーザーごとに個別の並び順を保存
- 「完了」ボタンで保存→通常モードに戻る

#### FR-014: チャネル表示フィルタ

- 施設詳細画面のチャネルタブは、`facility_accounts` 行があるチャネルのみ表示する
- マスタPWシートからチャネルを削除して同期すると、そのチャネルはUIから消える
- マスタに再追加して同期すれば復活する
- アカウントが1つも無い施設はエンプティステートを表示する

#### FR-015: チャネル単位の削除（admin限定）

- 施設詳細画面の各チャネルヘッダーに削除ボタン（赤いゴミ箱アイコン）を配置
- クリックで確認ダイアログ → DBとマスタPWシートの両方から削除
- リンカーンの場合はその施設×チャネルの全ユーザー行を一括削除
- シート削除を先に実行し、失敗時はDBを触らずエラーを返す（整合性保持）

### 2.3 マスタシート連携

#### FR-020: マスタ同期（シート→DB）

- Google Spreadsheetのアカウント情報をDBに取り込む
- 施設詳細画面の「マスタPWと同期」ボタンで実行
- パスワードは暗号化して保存
- チャネル固有の追加フィールド（楽天施設ID、ねっぱん契約コード等）も同期

#### FR-021: マスタ転記（DB→シート）

- DBのアカウント情報をGoogle Spreadsheetに書き戻す
- 施設詳細画面の「マスタに転記」ボタンで実行
- パスワードは復号して書き込み

#### FR-022: 一括同期での欠損検出

- 「全チャネル一括同期」実行時、DB に存在するがシートに無い `(facility, channel)` を検出
- 検出された場合、第2ダイアログでチェックボックスによる選択削除を促す
- ユーザーが「削除」を押した場合のみ DB から削除（シートは既に空のため触らない）
- 第2ダイアログのデフォルトは全選択。0件チェック時は削除ボタンを無効化

### 2.4 Chrome拡張

#### FR-030: ペアリング

- 拡張のポップアップからポータルURL + ペアリングコードを入力して接続
- デバイストークンで認証（以降のAPI通信に使用）

#### FR-031: 拡張デプロイ

- `pnpm deploy:extension` コマンドで自動ビルド→コピー→コミット→プッシュ
- 他PCでは `git pull` + 拡張リロードで更新

### 2.5 ヘルスチェック

#### FR-040: チャネルヘルスステータス

- チャネルごとの最終成功/失敗日時とエラーコードを記録
- ダッシュボードにステータスアイコンで表示

#### FR-041: ねっぱんPW経過日数アラート

- ねっぱん管理画面からPW変更アラートを取得・保存
- 施設詳細画面でアラート情報を表示

### 2.6 権限管理

#### FR-050: ロールベースアクセス制御

- admin: 全操作可能（施設作成・削除、アカウント編集）
- user: 閲覧 + ログイン実行 + 並べ替え + タグ編集
- 新規ユーザーはデフォルトでadmin（現行設定）

## 3. 非機能要件

### 3.1 セキュリティ

- NFR-001: パスワードはDBに暗号化して保存（`password_encrypted`）
- NFR-002: パスワードをログ・コンソール・成果物に出力しない
- NFR-003: API認証にSupabase Auth（Webポータル）+ デバイストークン（拡張）を使用
- NFR-004: RLS（Row Level Security）で全テーブルを保護
- NFR-005: `.env.local` をGit管理外に保持

### 3.2 可用性

- NFR-010: Vercel上で稼働（自動スケーリング）
- NFR-011: Supabase Cloudで稼働（マネージドDB）

### 3.3 拡張性

- NFR-020: チャネル追加は `channels.ts` + DB migration + マニフェスト更新で対応可能
- NFR-021: 追加フィールドは `account_field_definitions` で定義、チャネル固有の項目を柔軟に追加可能
